Introduction & Architecture Context

This document outlines the operational behavior, user interactions, technical requirements, implementation logic, and data flows for the Phase 1 MVP. The system utilizes a Flutter mobile app communicating via REST API to a Laravel backend hosted on AWS EC2. Data is stored in AWS RDS (PostgreSQL), and media is stored in Amazon S3. Web interfaces are server-rendered using Laravel Blade.

1

Employee Mobile App (Android)

1.1 Authentication & Module Selection Flutter / API

1. Operational Behavior

The app displays a splash screen, asks the user to select their division (NE Brothers or MAKU Security) for branding purposes, and presents a login form. Upon success, it establishes a persistent secure session.

2. User Interaction

User taps company logo → Enters Phone/Email & Password → Taps Login → Re-directed to Employee Dashboard.

3. Requirements

  • Internet connection.
  • Valid credentials assigned by Admin.

4. Implementation Strategy

Flutter captures credentials, POSTs to Laravel /api/login. Laravel validates against RDS, issues a Sanctum Personal Access Token (JWT), and returns it. Flutter stores the token in SecureStorage.

5. Data Flow

[User Input] → [Flutter POST /api/login] → [Laravel Controller] → [AWS RDS Auth Check] → [Laravel Issues Sanctum Token] → [Flutter SecureStorage]
1.2 Duty Check-In/Out (GPS + Selfie) Flutter / API

1. Operational Behavior

The core proof-of-presence mechanism. Prevents check-in if GPS or Camera access is denied. Calculates distance from assigned site coordinates (if required, or simply logs raw coordinates for Admin review).

2. User Interaction

User taps Start Duty → App requests Location/Camera permissions → User takes a selfie → App verifies location → Data is uploaded → Screen shows Duty Active. Same flow for End Duty (without selfie, just timestamp & GPS).

3. Requirements

  • Hardware GPS and Front-facing Camera.
  • OS-level permissions granted.
  • Active shift assigned in the database.

4. Implementation Strategy

Flutter uses geolocator and image_picker. Image is compressed via flutter_image_compress (target < 500kb). Multipart/form-data POST to /api/attendance/check-in containing {shift_id, lat, lng, image_file}. Laravel uses AWS SDK to stream file to S3, retrieves S3 URL, and writes attendance record to PostgreSQL RDS.

5. Data Flow

[Device GPS & Camera] → [Flutter Compression] → [POST + Bearer Token] → [Laravel] → [Upload to AWS S3] → [Save URL+Coords to AWS RDS]
1.3 View Assigned Shifts Flutter / API

1. Operational Behavior

Displays a list or calendar view of the employee's upcoming shifts, including site name, address, reporting time, and client contact.

2. User Interaction

User navigates to the My Shifts tab. Taps a specific shift to view site address. Can tap the address to open native Google Maps.

3. Requirements

  • Valid API Token.

4. Implementation Strategy

Flutter sends GET /api/shifts/me. Laravel fetches upcoming shifts from RDS where user_id = auth()->id(). Maps integration is done by passing coordinates to the url_launcher package (e.g., google.navigation:q=lat,lng), avoiding Google Maps API costs.

5. Data Flow

[Flutter GET /shifts] → [Laravel API] → [RDS Query] → [JSON Response] → [Flutter UI Render] → (Optional) [OS Native Map App]
1.4 Incident Reporting Flutter / API

1. Operational Behavior

Allows guards/workforce to log anomalies (theft, damage, absence, unauthorized entry) directly linked to their current site and shift.

2. User Interaction

User taps Report Incident → Selects Category (dropdown) → Types Description → Attaches Photo (optional) → Submits.

3. Requirements

  • Camera permission.

4. Implementation Strategy

Similar payload to Check-In. POST /api/incidents. Laravel stores the image in S3 (/incidents/{date}/file.jpg) and creates a database record tied to shift_id and site_id so Clients can view it later.

5. Data Flow

[User Form Data + Photo] → [Flutter POST] → [Laravel] → [AWS S3 (Image)] & [AWS RDS (Text)] → [Admin/Client Real-time View]
1.5 Emergency SOS Alert Flutter / API

1. Operational Behavior

A highly visible panic button that overrides normal reporting and immediately alerts the Admin panel with the user's last known exact GPS location.

2. User Interaction

User presses and holds SOS button for 3 seconds (to prevent accidental triggers) → Screen flashes red → Request sent.

3. Requirements

  • Immediate GPS access.

4. Implementation Strategy

POST /api/sos. Laravel writes a high-priority record to the incidents table. (In Phase 2, this would trigger an SMS. For Phase 1 MVP, it creates a flashing visual alert on the Admin Web Dashboard via polling).

Employee Web Portal (Browser)

2.1 Browser Authentication Laravel Blade

1. Operational Behavior

A fallback access point for employees whose Android devices are broken, lost, or incompatible. Accessible via a standard web URL.

2. User Interaction

User visits app.company.com/employee on their mobile browser (Chrome/Safari) → Logs in → Accesses mobile-optimized HTML interface.

3. Requirements

  • Web Browser.

4. Implementation Strategy

Standard Laravel session-based authentication (Laravel Breeze/Fortify). No API tokens needed here. Uses HTTP-only cookies.

2.2 Web-Based Duty Check-In (Geo/Cam) HTML5 / JS / Laravel

1. Operational Behavior

Replicates the Flutter app's check-in functionality using native browser APIs, ensuring proof of presence is still collected.

2. User Interaction

User taps Check In → Browser prompts Allow Location? → Browser prompts Allow Camera? → User takes photo using HTML video element → Submits.

3. Requirements

  • Browser permissions for navigator.geolocation and navigator.mediaDevices.getUserMedia.
  • HTTPS connection (required by browsers to allow camera/GPS access).

4. Implementation Strategy

JavaScript extracts Lat/Lng. JS captures a frame from the video stream, converts it to a Base64 string or Blob, appends it to a hidden form field, and submits a standard POST request to a Laravel Controller. Controller handles it identically to the API request.

5. Data Flow

[HTML5 Geo/Media API] → [JS Blob Creation] → [Laravel HTTP POST] → [AWS S3 & RDS]
3

Admin Web Panel

3.1 Master Dashboard & Attendance Monitor Laravel Blade

1. Operational Behavior

The command center for operations managers. Shows a high-level view of all active shifts, missing personnel, and recent incidents across all clients.

2. User Interaction

Admin logs in. The default view shows large statistic cards (Total Guards, On Duty, Absent) and a data table of today's attendance logs with click-to-view selfie links.

3. Requirements

  • Admin Role permission in DB.

4. Implementation Strategy

Laravel queries RDS attendance table joined with users and sites where date = today(). Uses a or simple JS polling to refresh the dashboard every 60 seconds without complex WebSockets (saving AWS compute).

3.2 Entity Management (Users, Clients, Sites) Laravel Blade

1. Operational Behavior

Standard CRUD (Create, Read, Update, Delete) interfaces to manage the backbone data of the system.

2. User Interaction

Admin clicks Clients → Add New. Fills out company name, contact info, and assigns a login password for the client. Admin clicks Sites → Adds location details linked to that Client.

3. Requirements

  • Relational integrity (Cannot delete a Client if Sites exist).

4. Implementation Strategy

Laravel Eloquent ORM handles relationships (Client hasMany Sites, Site hasMany Shifts). Blade templates render HTML forms. Laravel handles validation rules (e.g., unique email) before saving to RDS.

5. Data Flow

[Admin HTML Form] → [Laravel Controller Validation] → [Eloquent ORM] → [AWS RDS INSERT/UPDATE]
3.4 Daily Report Generation (PDF) Laravel / Snappy

1. Operational Behavior

Compiles a full day of activity for a specific site into a branded, printable PDF document containing shift timings, selfies, and incident notes.

2. User Interaction

Admin selects a Client → Selects a Site → Selects a Date → Clicks Generate Report. A PDF is downloaded to their local machine.

3. Requirements

  • Server-side PDF rendering engine.

4. Implementation Strategy

Laravel retrieves all data for the requested parameters. A special Blade template (HTML/CSS) is designed specifically for print format. Laravel passes this HTML to the barryvdh/laravel-snappy (wrapper for wkhtmltopdf) or dompdf package, which converts the HTML+Images into a downloadable PDF stream.

5. Data Flow

[Request Params] → [Fetch RDS Data & S3 Images] → [Compile Blade HTML] → [Snappy/DomPDF Engine] → [PDF Stream to Browser]
4

Client Web Panel

4.1 Traffic Light Dashboard (Restricted) Laravel Blade

1. Operational Behavior

A highly restricted, read-only view. Clients can only see data belonging to their specific client_id. Gives them peace of mind via the Traffic Light visual system.

2. User Interaction

Client logs in. Dashboard shows a list of assigned guards for today. Green indicator = On Duty. Yellow = Late. Red = Absent/Issue.

3. Requirements

  • Strict multi-tenant security logic to prevent data leaks between different clients.

4. Implementation Strategy

Laravel Global Scopes are applied to the Client role. E.g., Shift::where('client_id', auth()->user()->client_id). The Traffic Light logic is handled by Laravel Accessors: comparing shift.start_time vs attendance.check_in_time.

4.2 Site Incident Viewer & Report Access Laravel Blade

1. Operational Behavior

Allows the client to view anomalies reported by guards on their property, and download the exact same daily PDF reports that the Admin generates.

2. User Interaction

Client navigates to Incidents tab to see a feed of photos and notes. Navigates to Reports tab to click Download PDF for yesterday's shift.

3. Requirements

  • S3 URLs must be accessible to the browser to render images.

4. Implementation Strategy

To prevent hotlinking, images uploaded to S3 can be set as private, and Laravel generates temporary signed URLs valid for 60 minutes when rendering the Blade view. PDF generation uses the same Snappy logic as the Admin panel, but hard-codes the client_id parameter to the logged-in user.

5. Data Flow

[Client Request PDF] → [Laravel checks Client ID scope] → [Generate PDF via DomPDF] → [Download to Client Device]
